# Progress тАФ 2026-04-23

## Goal

Build the `earthcatalog` Python package: a single-node (and eventually
Dask-distributed) pipeline that reads an AWS S3 Inventory file (CSV, CSV.gz,
or Parquet), fetches STAC JSON files directly from a public S3 bucket via
obstore, transforms them into a spatially-partitioned GeoParquet table (H3
grid fan-out), and registers them in a PyIceberg catalog backed by a local
SQLite catalog. The `catalog.db` is shuttled to/from S3 with an atomic S3
lockfile.

## Session summary (2026-04-23)

### Problems identified & resolved

**1. `PARTITION_SPEC` incompatible with `add_files()`**

PyIceberg 0.11.0's `table.add_files()` requires each registered Parquet file
to contain exactly one value for any `IdentityTransform` partition field. Our
fan-out GeoParquet files span many H3 cells, causing:

```
ValueError: Cannot infer partition value from parquet metadata as there are
more than one partition values for Partition Field: grid_partition.
lower_value='820707fffffffff', upper_value='821937fffffffff'
```

**Fix**: Changed `PARTITION_SPEC = PartitionSpec()` (unpartitioned) in
`catalog.py`. Removed unused `PartitionField`, `IdentityTransform`, and
`YearTransform` imports. Spatial locality is already encoded in the GeoParquet
`geo` metadata and the `grid_partition` column.

**2. `schema.py` was orphaned тАФ not imported by any runtime code**

`transform.py` had inlined an equivalent `_COLUMNS` list (with an extra
`is_geometry` boolean needed to inject `geoarrow.wkb` metadata).
`catalog.py` defines its own `ICEBERG_SCHEMA` with stable PyIceberg field IDs.
Neither imported `schema.py`, making it dead documentation.

**Fix**: Added `GEOMETRY_FIELDS: frozenset[str]` to `schema.py`. Updated
`transform.py` to derive `_COLUMNS` from `schema.py` at import time:

```python
from .schema import GEOMETRY_FIELDS, schema as ARROW_SCHEMA

_COLUMNS = [
    (f.name, f.type, f.nullable, f.name in GEOMETRY_FIELDS)
    for f in ARROW_SCHEMA
]
```

Added a module-level `assert` that validates field order has not drifted.
Also corrected `geometry` nullability to `nullable=True` (rustac writes it as
optional; `catalog.py` already had `required=False`).

**3. No DuckDB query test existed**

Added `tests/test_duckdb_query.py` (7 tests) that verifies `duckdb.iceberg_scan()`
works against a freshly-built local catalog:
- row count > 0
- all source item IDs present
- `percent_valid_pixels` and `date_dt` readable as integers
- `platform` column non-null
- row count doubles after a second `add_files()` call
- SQL `WHERE` filter works
- `json_extract(raw_stac, '$.id')` works

**4. No explicit 0тЖТN insertion test existed**

`test_roundtrip.py::test_append_and_scan_row_count` already inferred 0тЖТN
structurally (fresh table тЖТ write тЖТ assert count), but never explicitly
asserted the pre-insert state. Added
`test_single_item_insert_zero_to_n` which:
1. Asserts `table.scan().to_arrow().num_rows == 0` before any write.
2. Calls `fan_out([ITEMS[0]], p)` + `write_geoparquet()` + `table.add_files()`.
3. Asserts `result.num_rows == n` and `result.column("id")[0].as_py() == ITEMS[0]["id"]`.
