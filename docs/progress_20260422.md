# Progress вЂ” 2026-04-22

## Goal

Build the `earthcatalog` Python package: a single-node (and eventually Dask-distributed) pipeline that reads an AWS S3 Inventory file (CSV, CSV.gz, or Parquet), fetches STAC JSON files directly from a public S3 bucket via obstore, transforms them into a spatially-partitioned GeoParquet table (H3 grid fan-out), and registers them in a PyIceberg catalog backed by a local SQLite catalog. The catalog.db is shuttled to/from S3 with an atomic S3 lockfile.

---

## Instructions

- Use **mamba** (not conda) for virtual environment commands. Environment name: `itslive-ingest`
- The S3 bucket `its-live-data` is **public** вЂ” use `obstore` with `skip_signature=True`, no AWS credentials needed
- **Never use boto3** вЂ” all S3 I/O goes through `obstore`
- STAC items live at prefix `test-space/velocity_image_pair/nisar/v02`, files end in `.stac.json`
- Float fields `date_dt` and `percent_valid_pixels` must be **rounded to int32**
- H3 partitioning uses **boundary-inclusive** cell assignment; **Points** use `h3.latlng_to_cell` directly
- Default H3 resolution is **3** for production; tests use **resolution 2**
- Colon characters in STAC property names are sanitized: `proj:code` в†’ `proj_code`, `sat:orbit_state` в†’ `sat_orbit_state`
- `pyiceberg-core` extra must be installed: `pip install "pyiceberg[pyiceberg-core]"`
- For the S3 lock, use **S3 Conditional Writes** (`If-None-Match: *` via `obstore.put(..., mode="create")`)
- All I/O goes through the injected store from `core/store_config.py`
- `obstore` raises `FileNotFoundError` (builtin) for missing keys
- **Do not use async/asyncio in the pipeline** вЂ” but `asyncio.new_event_loop().run_until_complete()` is acceptable inside a sync wrapper for rustac calls
- Use **rustac** (`rustac.GeoparquetWriter`) for writing GeoParquet output files вЂ” rustac handles all geo metadata (`geo` Parquet key, `geoarrow.wkb` extension on geometry column) automatically; never construct geo metadata manually
- Use **`table.add_files()`** (not `table.append()`) to register GeoParquet files into the Iceberg catalog
- Keep properties promoted to top-level columns (rustac does this automatically from STAC item properties)
- `S3Lock` must be held for the full duration of `run()` in `incremental.py`; `download_catalog` / `upload_catalog` are called inside the lock

## Discoveries

- `rustac.GeoparquetWriter.open()` is async вЂ” must be called with `asyncio.new_event_loop().run_until_complete()`; use a fresh loop to avoid conflicts
- rustac correctly promotes all STAC properties to top-level GeoParquet columns, including custom ones like `proj_code`, `grid_partition`, `raw_stac`
- rustac writes Python `int` values as Arrow `int64` (not float); Python `float` stays as `double`
- rustac writes standard STAC datetime fields (`datetime`, `start_datetime`) as `timestamp[ms, tz=UTC]`; non-standard ones like `mid_datetime` come out as `string`
- rustac writes `geo` key in Parquet file-level metadata and `ARROW:extension:name: geoarrow.wkb` on the geometry field вЂ” these come from rustac, not constructed manually
- When reading a rustac-written file with `pq.read_table()`, the file-level `geo` key does NOT automatically appear in `t.schema.metadata`; it must be read via `pq.ParquetFile(path).metadata.metadata.get(b'geo')` and then set on the Arrow schema via `schema.with_metadata({b'geo': geo_meta})`
- PyArrow `schema.with_metadata()` causes file-level Parquet metadata to be written via `pq.write_table()` вЂ” this is how the `geo` key is preserved in the aligned output file
- `table.add_files()` in PyIceberg 0.11.0 is available and works, but **requires each Parquet file to have only one value for any `IdentityTransform` partition field** вЂ” our fan-out GeoParquet files span many H3 cells, causing: `ValueError: Cannot infer partition value from parquet metadata as there are more than one partition values for Partition Field: grid_partition`
- **Fix applied**: Changed `PARTITION_SPEC` in `catalog.py` to `PartitionSpec()` (unpartitioned). The GeoParquet `geo` metadata already serves as the spatial index; Iceberg-level identity partitioning on `grid_partition` is incompatible with `add_files` on multi-cell files.
- `YearTransform` on `datetime` has the same multi-value issue вЂ” also removed by switching to unpartitioned spec
- `pq.write_table()` does NOT accept `custom_metadata` as a kwarg вЂ” metadata must go through Arrow schema: `schema.with_metadata({b'geo': ...})`
- The geometry field's `required=True` in the Iceberg schema also causes `add_files` schema mismatch (rustac writes geometry as optional); already fixed to `required=False`
- `_align_schema()` in `transform.py` handles: missing columns в†’ fill with `pa.nulls()`; type mismatches в†’ `col.cast(target, safe=False)`; `null`-typed columns в†’ `pa.nulls()` of correct type; geometry field в†’ annotate with `geoarrow.wkb` field metadata; forwards `geo` file-level metadata from rustac to output schema metadata
